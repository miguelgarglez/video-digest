# PRD: CLI Productization

Status: done
Category: enhancement

## Problem Statement

Miguel has a working local **Ingestion** path that turns one YouTube **Video** into a
structured **Digest**, and local execution avoids the cloud-IP transcript blocking
seen on Fly. The current CLI still feels like a development script: it is invoked via
`bun run`, has one mostly implicit command shape, and does not yet provide a stable
surface for agents, automations, or repeated daily terminal use.

Miguel wants the CLI to become a polished local product: installable on his Mac,
pleasant for humans, predictable for agents, and ready to become the foundation for
future Gmail delivery, playlist polling, and personal knowledge workflows.

## Solution

Refine the CLI into a local product command named `video-digest`.

The command should remain human-first by default: clear prompts, useful progress,
readable final output, and helpful failure messages. It should also expose an
agent-safe mode through stable commands, `--json`, documented exit codes, and
predictable artifact paths.

The first productized CLI slice will support:

```sh
video-digest ingest <youtube-url>
video-digest ingest <youtube-url> --email-preview
video-digest ingest <youtube-url> --json
video-digest doctor
video-digest list
video-digest open latest
```

Local installation comes before public publishing. The package should expose a stable
`bin` command for Miguel's Mac, but npm publication, release automation, and public
package support are out of scope for this PRD.

## User Stories

1. As Miguel, I want to run `video-digest ingest <url>`, so that the command reads like
   a product command instead of a repo script.
2. As Miguel, I want the existing `bun run video-digest <url>` behavior to keep working
   during migration, so that current muscle memory does not break immediately.
3. As Miguel, I want `video-digest` to be installable locally on my Mac, so that I can
   use it from any terminal directory.
4. As Miguel, I want the CLI to show polished progress in an interactive terminal, so
   that long transcript and digest steps feel alive rather than stuck.
5. As Miguel, I want non-interactive runs to avoid spinners, so that logs remain clean
   when agents or automations call the command.
6. As Miguel, I want `--json` output for ingestion, so that Codex or another agent can
   parse the result without scraping human text.
7. As Miguel, I want stable exit codes, so that automations can distinguish success,
   unsupported input, missing configuration, transcript unavailable, unusable
   transcript, and provider failure.
8. As Miguel, I want the final human output to include the **Digest** path, metadata
   path, transcript path, and optional email preview path, so that I can open the result
   quickly.
9. As Miguel, I want `video-digest doctor`, so that setup issues with Bun, uv, Python,
   OpenCode configuration, output directories, and transcript sidecar access are
   diagnosed before I run a real **Ingestion**.
10. As Miguel, I want `doctor` to explain what is missing and how to fix it, so that I
    learn the runtime shape of the tool instead of seeing a vague failure.
11. As Miguel, I want `doctor --json`, so that an agent can inspect readiness without
    relying on terminal copy.
12. As Miguel, I want `video-digest list`, so that I can see recent local **Digests**
    without remembering filenames.
13. As Miguel, I want `video-digest list --json`, so that agents can discover recent
    artifacts programmatically.
14. As Miguel, I want `video-digest open latest`, so that I can quickly open the newest
    **Digest** from the terminal.
15. As Miguel, I want `video-digest open <video-id>`, so that I can open a known
    **Digest** later.
16. As Miguel, I want failed opens to explain whether no **Digest** exists or the
    requested **Video** ID is unknown, so that I know whether to ingest again.
17. As Miguel, I want the CLI help text to list commands, options, environment
    variables, and examples, so that the command is self-documenting.
18. As Miguel, I want command names and output labels to use the project domain
    language, so that **Video**, **Ingestion**, **Transcript**, **Digest**, and
    **Artifact** stay consistent across docs and implementation.
19. As Miguel, I want the CLI internals split into testable modules, so that future
    Gmail delivery can reuse the same results without coupling to terminal rendering.
20. As Miguel, I want the first productization slice to avoid Gmail, playlist polling,
    daemons, and public publishing, so that the base CLI becomes excellent before new
    delivery channels are added.
21. As a Codex agent, I want to call `video-digest ingest <url> --json`, so that I can
    generate a **Digest** and reliably find its artifacts.
22. As a Codex agent, I want machine-readable errors, so that I can decide whether to
    ask Miguel for configuration, try another **Video**, or inspect provider output.
23. As a future automation, I want the CLI output contract to be stable, so that Gmail
    delivery can be added as an adapter instead of a rewrite.

## Implementation Decisions

- Keep TypeScript on Bun as the CLI implementation runtime.
- Keep the existing core **Ingestion** path behind `ingestVideo`; the CLI product work
  should improve command structure and adapters, not fork domain logic.
- Introduce an explicit command model with `ingest`, `doctor`, `list`, and `open`.
- Preserve backwards compatibility for the current single-URL invocation by mapping it
  to `ingest`.
- Add a stable `bin` surface named `video-digest` for local installation.
- Use local installation first. Do not publish to npm in this PRD.
- Treat human output and agent output as separate renderers over the same command
  results.
- Make `--json` imply non-interactive output: no spinner, no prompts, one JSON object
  written to stdout.
- Keep human progress behavior interactive only. Non-TTY output should remain plain and
  parseable enough for logs.
- Define a versioned JSON result shape for successful and failed command execution.
- Document and test stable exit codes.
- Implement `doctor` as a readiness report with individual checks and an aggregate
  pass/fail status.
- Keep provider network calls out of `doctor` by default; expensive or flaky live
  checks can be introduced later as an explicit option.
- Implement `list` and `open` over local artifacts and metadata already written by the
  output writer.
- Prefer opening Markdown **Digest** artifacts over metadata for human `open` commands.
- Use the operating system opener for `open` only in human mode. Agent mode should
  return paths instead of launching UI.
- Avoid new npm dependencies unless there is a clear need and Miguel explicitly
  approves installation.
- Keep real Gmail sending outside this PRD. The CLI should prepare the contracts that
  make Gmail delivery straightforward later.

## Testing Decisions

Tests should verify observable command behavior through public interfaces. They should
avoid checking private formatting helpers directly unless those helpers become explicit
rendering modules.

Test targets:

- Command parsing for `ingest`, legacy URL invocation, `doctor`, `list`, `open`, and
  shared flags.
- Human output for successful and failed `ingest` runs with fake **Transcript Source**
  and fake `Summarizer` behavior.
- JSON output contracts for `ingest --json`, `doctor --json`, and `list --json`.
- Stable exit codes for success, invalid URL, transcript unavailable, unusable
  transcript, missing configuration, and unexpected failure.
- `doctor` readiness checks with filesystem, environment, and command-runner fakes.
- Artifact discovery for `list` and `open` using temporary output directories.
- Backwards compatibility for `bun run video-digest <youtube-url>`.
- Help text behavior for no arguments, `--help`, and command-specific help if added.

Normal automated tests should not call YouTube, OpenCode, Gmail, or the user's real
global installation. A final manual smoke test may run one real **Ingestion** locally
after implementation.

## Out of Scope

- Real Gmail sending.
- Gmail draft creation.
- Recurring automations.
- Playlist polling.
- Local daemon or background worker.
- Cloud deployment changes.
- Web UI changes.
- Residential proxy configuration.
- Public npm publishing.
- Release automation.
- Signed binaries or standalone executable packaging.
- New transcript providers or audio transcription fallback.
- Multi-video **Insight** generation.

## Further Notes

This PRD deliberately recenters the product around a local CLI because local execution
uses Miguel's network environment, which currently avoids the transcript-provider
blocking seen from Fly. A refined CLI also gives agents a smaller and more reliable
surface than the web UI: agents can call a command, parse JSON, inspect artifacts, and
then compose follow-up actions such as Gmail delivery.

The long-term direction can still include web and mobile access, but the next best
investment is a durable local command contract. Once that contract is stable, Gmail,
playlist polling, local services, and public packaging become adapter decisions rather
than core rewrites.
