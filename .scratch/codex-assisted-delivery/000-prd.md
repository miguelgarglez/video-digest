# PRD: Codex-Assisted Playlist Delivery

Status: proposed  
Category: enhancement

## Problem Statement

Miguel has a local CLI that can turn a single YouTube **Video** into a **Digest** and **Artifact** outputs. The next useful product step is validating the full end-to-end workflow from private **Source Playlist** to personal email without building a production poller too early.

The workflow should let Codex use the in-app browser to read Miguel's `Resumir` playlist, select the newest **Video**, call the local CLI using agent-safe JSON output, and send the resulting digest through Gmail.

## Solution

Create a Codex-assisted workflow around the existing CLI:

```text
Source Playlist "Resumir"
-> Codex browser reads newest playlist row
-> normalize Video URL
-> video-digest ingest <url> --email-preview --json
-> Codex reads email preview
-> Gmail Delivery to Miguel
-> local processed Video state
```

This validates the user value of **Delivery** and helps identify gaps before implementing a robust programmatic **Trigger** with YouTube Data API polling.

## User Stories

1. As Miguel, I want to add a **Video** to the `Resumir` **Source Playlist**, so that it becomes eligible for digesting.
2. As Miguel, I want Codex to open the playlist directly by URL, so that the browser flow avoids manual navigation.
3. As Miguel, I want Codex to pick the newest playlist row, so that the workflow processes the intended next **Video**.
4. As Miguel, I want Codex to call `video-digest ingest --email-preview --json`, so that the workflow uses the stable CLI contract instead of terminal scraping.
5. As Miguel, I want the resulting digest emailed to myself, so that I can consume the output in my normal inbox.
6. As Miguel, I want duplicate detection, so that the same **Video** is not emailed repeatedly.
7. As Miguel, I want the workflow to support a premium transcript-only mode, so that Codex can author a richer digest when the **Video** deserves it.
8. As a future agent, I want playlist DOM extraction rules documented, so that I do not accidentally process recommendation links.
9. As a future automation, I want this workflow to stay separate from production polling, so that browser fragility does not leak into the core.

## Implementation Decisions

- Use the existing private playlist URL:

  ```text
  https://www.youtube.com/playlist?list=PLzfARYkgMjXqzneFV-50u3lYT8K7Mzvdw
  ```

- Treat the `list` ID as stable unless the playlist is deleted and recreated.
- Require playlist sort `Fecha de inclusión (más recientes)` for "newest first" behavior.
- Extract only from `ytd-playlist-video-renderer a#video-title`.
- Normalize extracted URLs to `https://www.youtube.com/watch?v=<videoId>`.
- Use `video-digest ingest <url> --email-preview --json` for baseline delivery.
- Use `video-digest transcript <url> --json` only for premium Codex-authored digests.
- Keep Gmail sending as a Codex app action, not a CLI feature in this PRD.
- Confirm exact Gmail recipient before first send because the conversation contains two similar addresses.
- Add local duplicate state before making this recurring.

## Testing Decisions

Initial verification can be manual because the browser and Gmail steps depend on connected Codex app state.

Test targets once code is added:

- URL normalization removes playlist and tracking parameters.
- Processed **Video** state prevents duplicate sends.
- Agent runbook uses CLI `--json` output only.
- Missing playlist rows are reported as "nothing to process".
- Missing email preview path fails before Gmail send.

## Out of Scope

- YouTube Data API polling.
- n8n workflow.
- Vercel Cron.
- Gmail implementation inside the CLI.
- Permanent cloud deployment.
- Processing multiple playlist videos per run.
- Removing videos from the playlist after processing.
- Publishing the CLI.
- Knowledge Base persistence beyond existing local **Artifacts**.
