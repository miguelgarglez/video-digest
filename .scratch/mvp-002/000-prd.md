# PRD: MVP 002 - Transcript Source Policy

Status: done  
Category: enhancement

## Problem Statement

Miguel needs the system to produce trustworthy **Digests** from YouTube **Transcripts**, but the current **Transcript Source** behavior hides important policy decisions. The language order is implicit, auto-generated transcript provenance is not represented, and **Transcript Quality** does not yet warn when a transcript arrives in a language outside the accepted policy.

Before expanding into playlist polling, real email, or a **Knowledge Base**, the transcript boundary should say clearly what transcript was obtained, why that transcript was chosen, and whether it is useful enough to summarize.

## Solution

Define and implement a narrow **Transcript Source** policy:

- Prefer `en` transcripts, then `es`.
- Do not translate transcripts automatically in the **Transcript Source** layer.
- Accept auto-generated transcripts, but capture **Transcript Provenance** when available.
- Keep `youtube-transcript-api` as the only automatic **Transcript Source** for now.
- Add deterministic **Transcript Quality** warnings for transcripts outside the language policy.

## User Stories

1. As Miguel, I want the transcript language preference to be explicit, so that future transcript bugs are easier to reason about.
2. As Miguel, I want the system to prefer English transcripts and then Spanish transcripts, so that current digest quality remains stable.
3. As Miguel, I want the system not to translate transcripts automatically, so that the saved **Transcript Artifact** remains close to the source evidence.
4. As Miguel, I want auto-generated transcripts to be accepted, so that useful videos are not blocked unnecessarily.
5. As Miguel, I want auto-generated transcript provenance to be stored when available, so that I can inspect whether a digest came from manual or generated text.
6. As Miguel, I want auto-generated provenance not to create a warning by itself, so that warnings stay tied to observed transcript usefulness.
7. As Miguel, I want unexpected transcript languages to produce a visible warning, so that I know when a digest may be less reliable.
8. As Miguel, I want low-quality transcripts to continue blocking LLM usage, so that I avoid paying for unreliable summaries.
9. As Miguel, I want warning-quality transcripts to continue generating **Digests**, so that I can decide whether the result is useful.
10. As Miguel, I want no automatic provider fallback yet, so that we learn from the first provider before adding more moving parts.
11. As Miguel, I want future providers to fit the same transcript contract, so that `yt-dlp`, SaaS, or ASR can be compared later.
12. As a future agent, I want the policy captured in the project docs, so that I do not infer behavior from hardcoded values.

## Implementation Decisions

- Add **Transcript Language Policy** and **Transcript Provenance** to the domain language.
- Keep the initial language policy as `en` then `es`.
- Keep translation out of the **Transcript Source** layer.
- Keep `youtube-transcript-api` as the only automatic provider in MVP 002.
- Extend the **Transcript** contract with provenance metadata that can represent unknown, manual, or auto-generated origin.
- Let **Transcript Quality** warn for languages outside policy.
- Do not make auto-generated transcripts a warning by default.
- Preserve `transcript.v0` and `transcript-quality.v0` unless a breaking schema change becomes necessary.

## Testing Decisions

Tests should verify observable transcript behavior through public interfaces.

Test targets:

- Python sidecar adapter preserves transcript provenance when the provider returns it.
- Python sidecar adapter accepts missing provenance as unknown metadata.
- **Transcript Quality** marks unsupported transcript languages as warning.
- **Transcript Quality** does not warn solely because a transcript is auto-generated.
- Existing transcript quality gates still block unusable transcripts.

External provider calls should not be used in normal tests. Use sidecar JSON fixtures and representative **Transcript** objects.

## Out of Scope

- Polling a **Source Playlist**.
- YouTube Data API integration.
- Adding `yt-dlp`, Supadata, TranscriptAPI, Apify, ASR, or AI multimodal fallback.
- Translating transcripts automatically.
- Real email sending.
- Changing **Digest** prompt structure.
- Benchmarking live transcript providers.

## Further Notes

This MVP deepens the transcript boundary before widening the product workflow. The goal is not to make transcript fetching perfect; it is to make the first transcript layer explicit, inspectable, and easy to compare against future **Fallback** options.
