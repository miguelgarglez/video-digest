# PRD: MVP 001 - Single Video Digest CLI

Status: ready-for-agent  
Category: enhancement

## Problem Statement

Miguel accumulates long YouTube videos and wants a reliable way to turn one selected **Video** into a useful **Digest** that can later feed a personal **Knowledge Base**. The first product milestone must prove the core path works before adding playlist polling, deployment, email sending, or multi-video synthesis.

## Solution

Build a local Bun/TypeScript CLI that accepts one YouTube URL, obtains a **Transcript** through the Python `youtube-transcript-api` sidecar, evaluates **Transcript Quality**, generates a structured **Digest** through OpenCode Zen, and writes local artifacts.

The CLI entrypoint is:

```sh
bun run video-digest <youtube-url> [--email-preview]
```

MVP 001 outputs:

```text
outputs/transcripts/<video-id>.json
outputs/digests/<video-id>.md
outputs/metadata/<video-id>.json
outputs/emails/<video-id>.md       # only with --email-preview
```

## User Stories

1. As Miguel, I want to run a local command with a YouTube URL, so that I can produce a useful **Digest** without manually opening YouTube.
2. As Miguel, I want the command to parse the **Video** ID from common YouTube URL formats, so that I can paste links naturally.
3. As Miguel, I want the system to obtain a **Transcript** automatically, so that I do not have to copy text from YouTube.
4. As Miguel, I want the full **Transcript** saved as a **Transcript Artifact**, so that I can debug or compare the **Digest** against the source text.
5. As Miguel, I want a **Transcript Quality** status, so that I can see whether the transcript was good enough to trust.
6. As Miguel, I want low-quality transcripts to stop before LLM usage, so that I do not spend money on unreliable summaries.
7. As Miguel, I want warning-quality transcripts to still produce a **Digest** with visible warnings, so that I can decide whether the result is useful.
8. As Miguel, I want OpenCode Zen to generate a structured **Digest**, so that I can consume the video quickly.
9. As Miguel, I want a generated **Digest Title**, so that the note has a useful title even when YouTube metadata is unavailable.
10. As Miguel, I want Markdown output, so that I can read the **Digest** directly.
11. As Miguel, I want JSON metadata output, so that future tools can consume the result programmatically.
12. As Miguel, I want an optional email preview, so that we can validate the future email format without adding real email delivery.
13. As Miguel, I want secrets to live in `.env`, so that API keys are not committed.
14. As Miguel, I want clear errors when configuration is missing, so that setup problems are obvious.
15. As Miguel, I want provider logic isolated behind interfaces, so that we can later swap transcript or LLM providers.
16. As Miguel, I want the core to stay reasonably portable, so that future Vercel, Node, or frontend work remains possible.

## Implementation Decisions

- Use TypeScript on Bun 1.3.14 for the core, CLI, and tests.
- Use `uv` for the Python sidecar that wraps `youtube-transcript-api`.
- Keep Python usage behind a **Transcript Source** adapter.
- Use only `youtube-transcript-api` as the initial **Transcript Source**. MVP 001 does not include `yt-dlp`, Supadata, TranscriptAPI, Apify, or ASR fallback.
- Introduce a `Summarizer` interface from the first MVP.
- Implement `OpenCodeSummarizer` as the first `Summarizer` adapter.
- Configure OpenCode through `.env` using `OPENCODE_API_KEY`, `OPENCODE_BASE_URL`, and `OPENCODE_MODEL`.
- Default `OPENCODE_MODEL` to `gpt-5-nano`.
- Do not use YouTube Data API in MVP 001.
- Store factual **Video** metadata as `null` when unavailable.
- Generate a **Digest Title** through the `Summarizer`; do not treat it as the original video title.
- Store full transcripts as local **Transcript Artifacts**, not as **Knowledge Items**.
- Evaluate **Transcript Quality** with deterministic `transcript-quality.v0` heuristics.
- Treat `usable` and `warning` transcripts as summarizable.
- Treat `unusable` transcripts as blocking: do not call OpenCode and exit with code 2.
- Generate email preview files only when `--email-preview` is provided.
- Do not send real email in MVP 001.

## Testing Decisions

Tests should verify observable behavior through public interfaces, not private implementation details.

Test targets:

- YouTube URL parsing into canonical `videoId`.
- **Transcript Quality** classification from representative transcript fixtures.
- **Digest** schema validation and serialization.
- CLI behavior for missing arguments and missing OpenCode config.
- Ingestion orchestration with fake **Transcript Source** and fake `Summarizer`.
- Output writers creating the expected file set.

External provider calls should not be used in normal tests. Use fakes/fixtures for `TranscriptSource` and `Summarizer`, with one manual smoke path reserved for real provider testing.

## Out of Scope

- Polling a **Source Playlist**.
- YouTube Data API integration.
- n8n integration.
- Vercel deployment.
- Frontend UI.
- Real email sending.
- Multiple transcript fallbacks.
- ASR or multimodal video analysis.
- Multi-video **Insight** generation.
- Obsidian, Notion, Drive, or Readwise export.
- Public npm or `npx` packaging.

## Further Notes

The MVP should optimize for learning and correctness over completeness. The most important proof is that one **Video** can become a trustworthy **Digest** with clear artifacts, clear failure modes, and provider seams that do not leak into the domain core.
