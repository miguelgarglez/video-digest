# PRD: CLI Interactive Configuration

Status: done
Category: enhancement

## Problem Statement

The productized CLI now has full **Digest** mode and token-free transcript-only mode,
but the interactive flow still behaves like the original MVP. It asks for a URL and
email preview, then discovers missing `OPENCODE_API_KEY` late during digest
generation.

Miguel wants the interactive CLI to guide users through the right path: choose full
**Digest** or transcript-only, securely configure an OpenCode token when needed, and
continue without forcing manual shell setup.

## Solution

Add secure local credential storage and improve the interactive CLI flow.

On macOS, the OpenCode API key should be stored in Keychain as a generic password:

```text
service: personal-video-digest
account: opencode-api-key
```

The CLI resolves OpenCode credentials in this order:

1. `OPENCODE_API_KEY` environment variable.
2. macOS Keychain.
3. Not configured.

Add non-interactive config commands:

```sh
video-digest config set opencode-api-key
video-digest config get
video-digest config unset opencode-api-key
```

Improve `video-digest` with no args:

```text
What do you want to create?
  1. Digest
  2. Transcript only

Choice [1]:
YouTube URL:
```

If **Digest** is selected and no token is configured, offer a guided setup with an
OpenCode link, paste-token prompt, optional Keychain persistence, and then continue
the digest in the same run. If the user declines to configure a token, offer
transcript-only fallback.

## User Stories

1. As Miguel, I want interactive mode to ask whether I want a **Digest** or transcript
   only, so that the CLI matches the product capabilities.
2. As Miguel, I want the CLI to detect whether OpenCode is configured before starting
   digest generation, so that it does not fail late.
3. As Miguel, I want the CLI to show where to get an OpenCode API key, so that setup
   is discoverable.
4. As Miguel, I want to paste an OpenCode API key during the interactive flow, so that
   I can continue without editing shell files.
5. As Miguel, I want the CLI to save the key securely in macOS Keychain, so that future
   global runs work from any directory.
6. As Miguel, I want to choose transcript-only fallback if I do not want to configure a
   key, so that the command still produces useful artifacts.
7. As Miguel, I want `video-digest config get`, so that I can see whether digest
   credentials are configured without printing the secret.
8. As Miguel, I want `video-digest config unset opencode-api-key`, so that I can remove
   the stored credential.
9. As an agent, I want `--json` commands to remain non-interactive, so that automation
   never hangs waiting for a token.

## Implementation Decisions

- Add a `CredentialStore` interface with a macOS Keychain implementation.
- Use the macOS `security` CLI for Keychain access.
- Never print the token after capture.
- Do not store the token in the repository `.env`, `~/.zshrc`, or a plain JSON config
  file.
- Keep `OPENCODE_API_KEY` as the highest-priority credential source.
- Inject the resolved token into `OpenCodeSummarizer` instead of mutating
  `process.env`.
- Add `config` to the CLI command model.
- Keep `config set opencode-api-key` interactive only; no `--value` flag in this slice.
- Keep tests isolated with fake credential stores and fake command runners. Automated
  tests must not read or write Miguel's real Keychain.

## Testing Decisions

Tests should verify behavior through public CLI flows and a fake credential store.

Test targets:

- Keychain command construction for get/set/unset.
- Credential resolution priority: env before Keychain.
- `config get` reports configured source without printing secrets.
- `config set opencode-api-key` stores a prompted token.
- `config unset opencode-api-key` deletes the stored token.
- Interactive mode chooses **Digest** or transcript-only.
- Interactive digest setup can store a token and continue ingestion.
- Interactive digest setup can decline token setup and fallback to transcript-only.
- JSON mode remains non-interactive.

## Out of Scope

- Cross-platform credential stores.
- Plaintext config-file fallback.
- Browser-opening the OpenCode link.
- Token validation against OpenCode during setup.
- Public npm publishing.
- Gmail delivery.

## Further Notes

The implementation should make future cross-platform support straightforward by
keeping Keychain behind the `CredentialStore` interface.
